## 왜 다들 M4 맥미니를 이야기하는가

OpenClaw 이야기가 나오면 유난히 `M4 맥미니`가 자주 따라붙는다. 처음에는 단순히 성능이 좋아서 그런가 싶었는데, 공식 문서를 읽고 실제 설치 흐름을 따라가 보니 이유는 조금 달랐다. 이 조합이 자주 언급되는 핵심은 계산 성능 하나보다는, **macOS에서 OpenClaw를 훨씬 자연스럽게 붙여 쓸 수 있는 구조**에 있었다.

OpenClaw 쪽 문서를 보면 온보딩 경로가 크게 둘로 나뉜다. 하나는 macOS, Linux, Windows에서 공통으로 쓰는 CLI 기반 게이트웨이 경로이고, 다른 하나는 macOS 전용 companion app 경로다. 이 macOS 앱은 단순한 실행기가 아니라 메뉴바 앱, 권한 관리자, 게이트웨이 브로커 역할을 같이 맡고 있다. 문서상으로도 이 앱은 알림, Accessibility, Screen Recording, Microphone, Speech Recognition, Automation 같은 TCC 권한을 다루고, macOS 전용 도구를 노출하는 쪽에 가깝다.

여기에 iMessage 계열 채널까지 얹히면 이야기가 더 분명해진다. OpenClaw 문서는 새 iMessage 배포에는 BlueBubbles를 권장하고 있고, 기존 `imsg` 경로도 결국 Mac 환경을 전제로 한다. 그러니 사람들이 M4 맥미니를 이야기할 때 실제로 기대하는 것은 “작고 꽤 괜찮은 Apple 실리콘 머신”이면서 동시에 “macOS 전용 채널과 개인 비서 경험까지 한 번에 얹기 좋은 상자”에 더 가깝다.

하드웨어 조건도 이 인식을 밀어 준다. 현재 Mac mini의 M4 기본 구성이 16GB unified memory에서 시작하고, 공식 스펙 기준 메모리 대역폭이 120GB/s다. 이 수치 자체가 곧바로 OpenClaw 게이트웨이에 필요하다는 뜻은 아니지만, 적어도 사람들이 “나중에 더 많은 걸 붙일 수 있는 여유가 있다”고 느끼기에는 충분한 조건이다.

## 굳이 N100 + Ubuntu로 해봤다

그렇다고 해서 OpenClaw를 쓰려면 곧바로 M4 맥미니를 사야 하는가 하면, 그건 또 아니라고 느꼈다. 이유는 비교 범위를 어디에 두느냐에 따라 달라지기 때문이다. 이번에 내가 확인하고 싶었던 것은 `로컬 모델 추론 장비`가 아니라, **클라우드 모델과 연결되는 OpenClaw 게이트웨이**를 어떤 장비에서 안정적으로 굴릴 수 있는가였다.

이 관점으로 보면 하드웨어에 요구되는 조건이 꽤 달라진다. 이 경우 병목은 GPU나 메모리 대역폭보다는, 오래 켜 둘 수 있는지, 설치와 재설치가 단순한지, 원격 관리가 쉬운지, 비용이 과하지 않은지 쪽으로 이동한다. OpenClaw 공식 문서도 Linux 게이트웨이를 정식 지원하고, Node 22 이상만 있으면 설치 스크립트와 온보딩 wizard로 빠르게 올릴 수 있는 쪽에 가깝다.

Intel N100 기반 미니 PC는 바로 이 지점에서 꽤 그럴듯한 후보가 된다. 공식 스펙 기준으로 4코어 4스레드, 최대 16GB 메모리, 1채널 메모리 구성이며, TDP도 낮은 편이다. 로컬 LLM을 밀어붙일 장비로 보면 분명 아쉬운 조건이지만, 클라우드 모델만 붙이는 게이트웨이로 보면 오히려 성격이 또렷하다. 고성능 워크스테이션이 아니라, 전기 덜 먹고 계속 켜 두기 쉬운 작은 서버처럼 쓰기 좋다.

그래서 이번 실험은 “맥미니의 대체재를 찾는다”기보다, “OpenClaw에서 정말 비싼 하드웨어가 필요한 구간이 어디인지”를 분리해 보는 쪽에 가까웠다.

## Ubuntu에 올려 보니 병목은 하드웨어보다 운영 방식이었다

OpenClaw 설치 자체는 macOS와 Linux 공통으로 installer script와 onboarding wizard가 준비되어 있고, Linux에서는 systemd user service 기준으로 계속 켜 두는 흐름도 문서화되어 있다.

실제로 체감상 쉬웠던 부분은 세 가지였다. 첫째, Linux 서버처럼 다루면 된다는 점이다. Node 버전만 맞추고 게이트웨이를 올린 뒤 대시보드나 브라우저 Control UI로 붙으면 되니, 로컬 데스크톱 앱을 억지로 맞추는 느낌이 없다. 둘째, 원격 관리가 쉽다. SSH, systemd, 재설치 같은 익숙한 도구로 대부분 해결할 수 있어서 “서비스를 유지한다”는 감각에 가깝다. 셋째, 클라우드 provider만 붙이면 하드웨어 부담이 크지 않다. 모델 추론은 OpenAI나 Anthropic 같은 외부 provider가 맡고, 로컬 머신은 게이트웨이와 채널, 설정을 유지하는 쪽이 중심이 된다.

반대로 불편했던 점도 분명했다. 가장 큰 차이는 **Ubuntu 쪽은 OpenClaw를 개인 비서 앱이라기보다 셀프호스팅 서비스처럼 쓰게 된다**는 점이다. macOS 앱이 제공하는 권한 통합, 메뉴바 경험, macOS 전용 도구, iMessage/BlueBubbles 같은 경로는 Linux 박스만으로는 자연스럽게 따라오지 않는다. 같은 OpenClaw라도 M4 맥미니 쪽은 “책상 위에 두는 비서 장치”에 가깝고, Ubuntu/N100 쪽은 “작은 게이트웨이 서버”에 가깝다.

손해도 있다. 나중에 욕심이 생겨서 Ollama 같은 로컬 모델까지 한 박스에서 돌리고 싶어지면, N100의 한계는 금방 드러난다. 16GB 상한, 1채널 메모리, 저전력 CPU 조합은 게이트웨이에는 맞지만 로컬 추론 장비에는 넉넉하지 않다. 그러니 이번 비교에서 `로컬 모델`을 핵심 결론에서 뺀 이유도 여기 있다. 비교 범위를 넓히는 순간 결론 자체가 달라지기 때문이다.

## 결국 어떤 쪽이 더 맞는가

이번에 정리한 결론은 생각보다 단순하다. **클라우드 모델만 쓰는 OpenClaw 게이트웨이 기준이라면, M4 맥미니가 필수라고 보기는 어렵다.** Intel N100 미니 PC에 Ubuntu를 올려도 충분히 실용적이다. 이 조합은 저전력, 저비용, 재설치 용이성, 홈서버 같은 운영 방식에서 꽤 좋은 균형을 보여 준다.

다만 M4 맥미니가 더 자연스러운 경우도 분명하다. OpenClaw를 단순 게이트웨이로 쓰는 게 아니라, macOS 앱과 알림, 음성 입출력, BlueBubbles/iMessage 같은 채널, 항상 켜 둔 개인 비서 경험까지 한 번에 기대한다면 Mac 쪽이 훨씬 매끈하다. “OpenClaw를 돌린다”가 아니라 “OpenClaw를 내 생활 환경 안에 붙인다”는 쪽으로 갈수록 M4 맥미니의 장점이 커진다.

반대로 목적이 더 좁고 분명하다면 Ubuntu/N100도 충분히 납득 가능한 선택이다. 특히 게이트웨이를 조용히 한쪽에 두고, 클라우드 모델과 연결해 안정적으로 굴리고, 문제가 생기면 Linux 서버 다루듯 관리하고 싶은 쪽이라면 오히려 이 조합이 더 현실적일 수 있다.

결국 핵심은 하드웨어 이름이 아니라 용도였다. OpenClaw가 `개인 비서 경험`에 가까운지, 아니면 `항상 켜 두는 게이트웨이`에 가까운지에 따라 정답이 달라진다. 이번에 Ubuntu/N100으로 직접 올려 본 뒤 남은 판단은, 적어도 후자 쪽이라면 M4 맥미니는 좋은 선택일 수는 있어도 필수 선택은 아니라는 것이다.
