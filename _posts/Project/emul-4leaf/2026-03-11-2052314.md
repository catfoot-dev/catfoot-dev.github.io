## 범용 에뮬레이터보다 부팅 경로부터

`emul-4leaf`에서 먼저 붙잡은 것은 “윈도우 전체를 흉내내는 환경”이 아니라 `4Leaf` 클라이언트가 실제로 기대하는 시작 경로였다. 지금 구조는 EXE 하나를 통째로 띄우는 구조가 아니다. `Resources` 아래에 둔 `Core.dll`, `WinCore.dll`, `DNet.dll`, `Lime.dll`, `Rare.dll`, `4Leaf.dll`을 순서대로 메모리에 올리고, 재배치와 import 해석을 거친 뒤 마지막에 `4Leaf.dll!Main`을 직접 호출하는 부팅 루프이다.

겉으로는 크로스 플랫폼 목표를 이야기하고 있지만, 안쪽은 당분간 32비트 `Win32` 가정을 전제로 움직인다. 이 프로젝트에서 더 중요했던 것은 일반적인 호환 계층을 한 번에 완성하는 일이 아니라, 특정 클라이언트가 어떤 주소 배치와 API 호출을 기대하는지 하나씩 드러나게 만드는 일이었기 때문이다. 범위를 좁혀 둔 덕분에 적어도 지금은 “무엇을 흉내내야 다음 단계로 넘어가는가”가 보이기 시작했다.

## Win32를 전부 만들지 않고 필요한 면만 받치기

이 프로젝트가 마음에 드는 지점은, 운영체제를 다 구현하려고 들기보다 필요한 경계부터 세운다는 점이다. import를 해석할 때 이미 로드된 DLL export가 있으면 그 주소로 바로 연결하고, 없으면 `FAKE_IMPORT_BASE` 영역에 가짜 주소를 할당한 뒤 `KERNEL32.dll`, `USER32.dll`, `WS2_32.dll` 같은 핸들러로 넘긴다. 결국 중요한 것은 API 이름을 아는 일이 아니다. 어느 호출이 실제 실행 흐름을 막고 있는지 식별할 수 있는 통로를 만드는 일이다.

이 방식 덕분에 지금 구현은 꽤 거칠어도 앞으로 채워 넣을 자리가 분명하다. 어떤 함수는 실제 export를 타고 들어가고, 어떤 함수는 Rust 쪽 핸들러에서 받아 주고, 또 어떤 함수는 이름만 남긴 채 스텁으로 지나간다. 곧바로 완성된 것은 아니다. 오히려 `USER32` 나 `WS2_32` 쪽에는 호출 사실만 기록하고 넘어가는 함수가 아직 많다. 그래도 이 편이 처음부터 정교한 호환성을 목표로 잡는 것보다 낫다고 봤다. 적어도 지금은 어디가 비어 있는지 숨기지 않고 드러내는 구조이기 때문이다.

## 보이지 않던 흐름을 밖으로 꺼내기

에뮬레이터 쪽 작업은 실행되는 것만큼 보이는 것이 중요했다. stdout 로그만 붙잡고 있으면 레지스터가 언제 어긋났는지, 스택이 어디서부터 흐트러졌는지, 지금 막 밟으려는 명령어가 무엇인지 따라가기 어렵다. 그래서 이번에는 코드 훅을 전체 실행 구간에 걸고, 매 스텝마다 레지스터 9개와 스택 상단, 다음 명령 바이트를 `CpuContext`로 모아 별도 디버그 창에 보내는 구조를 붙였다. `F10`으로 한 스텝씩 넘기게 만든 것도 같은 이유다. 이 프로젝트의 디버거는 부가 기능이 아니다. 실행 흐름을 붙잡아 두는 최소 장비이다.

내장 서버도 같은 맥락에서 들어갔다. `0.0.0.0:33000`에 서버를 띄우고 콘솔에서 HEX 문자열을 넣어 연결된 클라이언트에 브로드캐스트할 수 있게 해 둔 덕분에, 외부 서버를 따로 맞추지 않아도 패킷 흐름을 확인할 수 있게 됐다. 아직 완전한 프로토콜 재현이라고 보기는 어렵지만, 적어도 “연결은 어디서 열리고 어떤 데이터가 오가나”를 손 안에서 확인하는 데에는 충분한 시작점이 됐다.

## 아직 거친 부분

지금 기준의 `emul-4leaf`는 완성도보다 관찰 가능성 쪽에 더 무게가 실려 있다. `EIP`가 0으로 떨어지는 상황을 따로 잡아 두고, unmapped memory 접근을 훅으로 막고, 호출 규약이 어긋나 스택 포인터가 밀리는 경우를 로그로 확인하는 흐름이 그 사실을 잘 보여 준다. 돌아간다고 해도 제대로 돌아간 것인지 확신하기 어려운 이유가 여기에 있다. 스텁이 많다는 것은 곧 “성공처럼 보이는 실패”가 아직 섞여 있을 수 있다는 뜻이기도 하다.

그래도 이번 구조에서 얻은 것은 분명하다. 이제는 막연히 윈도우 API를 흉내내는 프로젝트가 아니다. DLL 로딩, import 해석, API 핸들링, 네트워크 관찰이라는 네 개의 경계로 쪼개서 볼 수 있는 프로젝트가 됐다. 다음 단계의 핵심은 지금 비워 둔 API를 실제 의미 있는 동작으로 바꾸고, 패킷 흐름과 UI 쪽에서 정말 필요한 `Win32` 의미론만 남기는 일이다. 그 작업이 쌓여야 비로소 “실행된다”에서 “믿고 다룰 수 있다” 쪽으로 넘어갈 수 있을 것 같다.
